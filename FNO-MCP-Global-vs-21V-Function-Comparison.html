Why can't VS Code connect to 21V directly?

1. Different Identity Provider

21V uses login.partner.microsoftonline.cn (China sovereign cloud). VS Code's MCP HTTP client is hardcoded for the global login.microsoftonline.com endpoint — no config option exists for alternate authorities.

2. Broken OAuth Metadata

The 21V MCP endpoint returns malformed RFC 9728 OAuth protected-resource metadata. VS Code's proxy JavaScript crashes with code?.toLowerCase is not a function when it processes the 21V response.

3. No Sovereign Cloud Config

VS Code's MCP server config schema has no authority or cloud setting to redirect token acquisition. The bridge script works around this by performing device-code auth itself, then feeds the token to a local stdio-to-HTTP proxy.

Recommendations

  • Functional parity confirmed: All 21 MCP tools work identically on both Global and 21V environments. No tool-level differences detected.
  • 21V connectivity: Continue using the bridge script (dynamicsfo-21v-mcp-bridge.ps1) until VS Code adds sovereign cloud OAuth support.
  • Prefer data tools for CRUD; use form tools for complex workflows (journal posting, approval flows) and for browsing large entity sets.
  • Token management: The bridge caches refresh tokens locally; re-authentication needed only when the refresh token expires.
  • ResponseTooLarge: Use precise $filter queries and limit $select fields to keep responses under 160 KB.
